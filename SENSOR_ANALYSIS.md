# Анализ сенсорной подсистемы проекта

## Обзор
Система использует датчики температуры DS18B20 на шине OneWire для мониторинга различных точек в дистилляционной колонне. Поддерживается до 4 датчиков на одной шине.

## Основные компоненты

### 1. DS18B20 Manager (`ds18b20_manager.h/cpp`)
- **Singleton паттерн** для управления всеми датчиками
- **Неблокирующий опрос** через конечный автомат с состояниями:
  - IDLE → REQUEST → WAITING → READING → IDLE
- **Асинхронный режим** работы DallasTemperature (`setWaitForConversion(false)`)
- **Автоматическое сканирование** шины при инициализации
- **Сопоставление** обнаруженных датчиков с конфигурацией по адресу
- **Поддержка калибровки** (автоматическая и ручная)
- **Максимум 4 датчика** (ограничение `MAX_SENSORS` в config.h)

### 2. Калибровка (`calibration.h/cpp`)
- **Алгоритм среднего значения**: Все калиброванные датчики должны показывать одинаковую температуру в одной среде
- **Процесс**:
  1. Сбор N отсчётов (CALIBRATION_SAMPLES = 20) с интервалом CALIBRATION_INTERVAL = 200мс
  2. Вычисление среднего для каждого датчика
  3. Нахождение глобального среднего
  4. Вычисление смещения: offset[i] = global_mean - average[i]
- **Хранение смещений** в SensorData.offset и сохранение в settings.json
- **Применение коррекции**: T_corrected = T_raw + offset (если calibrate=true) или T_corrected = T_raw + manual_offset

### 3. Конфигурация датчиков (config.h)
- **SensorRole enum**: Определяет роли датчиков в системе:
  - column_top - верх колонны
  - head_selection - узел отбора головных фракций
  - body_selection - узел отбора тела
  - cooler - температура охлаждающей жидкости
  - boiler - температура пара в кубе (для дистилляции)
  - custom - пользовательская роль
- **SystemConfig structure**: Содержит массив SensorData[MAX_SENSORS] с настройками для каждого датчика:
  - address_hex - HEX-строка адреса датчика
  - role - роль датчика
  - name - человекочитаемое имя
  - calibrate - флаг автоматической калибровки
  - offset - смещение калибровки
  - manual_offset - ручное смещение
  - active - флаг активности

## Потенциальные проблемы и замечания

### Технические ограничения
1. **Ограничение на 4 датчика** жестко задано в `MAX_SENSORS = 4` - нельзя подключить больше без перекомпиляции
2. **Одна шина OneWire** - все датчики должны быть подключены к одному GPIO пину
3. **Время конвертации** - 12-bit разрешение требует ~750мс, опрос происходит каждую секунду (SENSOR_POLL_MS = 1000)

### Калибровка
1. **Блокировка во время калибровки** - в функции `pollCalibration()` используется `delay(800)`, что блокирует выполнение
2. **Требование одинаковых условий** - для корректной калибровки все датчики с calibrate=true должны быть в одной среде
3. **Зависимость от порядка** - алгоритм калибровки чувствителен к порядку датчиков в массиве

### Обработка ошибок
1. **Отсутствие датчиков** - при не обнаружении датчиков функция `begin()` возвращает false, но система может продолжить работу
2. **Отключенные датчики** - обрабатываются через проверку на `DEVICE_DISCONNECTED (-127.0f)`
3. **CRC проверка** - при сканировании шины выполняется проверка CRC для исключения ложных срабатываний

### Производительность
1. **Неблокирующий дизайн** - основной опрос не блокирует выполнение loop()
2. **PSRAM не используется** - для хранения данных датчиков используется обычная память (SensorData sensors_[MAX_SENSORS])
3. **Статическое выделение памяти** - все буферы имеют фиксированный размер, нет динамического выделения

## Рекомендации по улучшению

1. **Сделать количество датчиков настраиваемым** через конфигурацию вместо жесткого лимита
2. **Поддержка множества шин OneWire** для большей гибкости в подключении датчиков
3. **Улучшить обработку ошибок** - добавить механизм восстановления при временных сбоях датчиков
4. **Оптимизировать калибровку** - убрать блокирующие вызовы delay() во время калибровки
5. **Добавить диагностические функции** - статистику по успешным/неудачным чтениям, среднее время отклика и т.д.
6. **Расширить роли датчиков** - добавить дополнительные роли для будущих расширений функционала
7. **Валидация конфигурации** - проверять соответствие настроенных ролей датчиков выбранному режиму работы

## Заключение
Сенсорная подсистема реалирована с учетом ограничений встраиваемых систем:
- Использует шаблон Singleton для глобального доступа
- Реализует неблокирующий опрос через конечный автомат
- Поддерживает автоматическую и ручную калибровку
- Имеет четкое разделение ответственности между модулями
- Слежит за ограничениями памяти (статическое выделение, отсутствие динамического allocation)
- Предоставляет гибкую систему ролей датчиков для разных режимов работы

Код хорошо структурирован и следует принятым в проекте конвенциям стиля.
